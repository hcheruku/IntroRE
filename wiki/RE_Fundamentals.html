Throughout the history of invention curious minds have sought to understand the inner workings of their gadgets. Whether investigating a broken watch, or improving an engine, these people have broken down their goods into their elemental parts to understand how they work. This is Reverse Engineering (RE), and it is done every day from recreating outdated and incompatible software, understanding malicious code, or exploiting weaknesses in software.

In this course we will explore what drives people to reverse engineer software and the methodology and tools used to do it.

!Input Processing
At the start of this phase we see the input passed to another function along with a pointer to  local variable var_1C. We know this is a pointer because of the lea before the push:
{{{
lea     eax, [ebp+var_1C]
push    eax
}}}
Jumping to this function, we find that a call to sscanf and a potential call to set off the bomb. This condition that leads to failure is:
{{{
mov     [ebp+var_4], eax
cmp     [ebp+var_4], 6
jge     short loc_11116FB
}}}
where eax is returned from sscanf. In other words, jump if sscanf returned 6 or greater. sscanf takes our input string, a format string ("%d %d %d %d %d %d"), and a series of 6 offsets from our pointer passed from the caller. It returns the number of items succesfully read. This tells us that we need to give this phase 6 integers, separated by spaces, and that that this functions second argument is a pointer to an array of at least 6 integers.
!The loop
After returning from the input scanner, there is a check to ensure the first item in the list is 1. The remainder of the function consists of a loop and return block. The loop is a standard ''for loop'' which uses var_4 as it's counter.
{{{
for( var_4 = 1; var_4 <= 6; var_4++) {...}
}}}
Each item in the list is compared with eax, which gets calculated like so:
{{{
mov     eax, [ebp+var_4]
add     eax, 1
mov     ecx, [ebp+var_4]
imul    eax, [ebp+ecx*4+var_20]
}}}
The final instruction can be a bit confusing. It references a variable that isn't actually allocated (var_20). This is a result of starting the counter at one instead of zero. This instruction has to be able to access the "0th" element of the array. A better representation for this instruction might be:
{{{
imul    eax, [ebp+var_1C+ecx*4-4]
}}}
This can help clarify that we are using an offset from the list (ebp+var_1C) in intervals of 4 (the size of each integer).  You can tell IDA to display the operand this way by clicking on the operand and pressing Alt+F1 (Manual...). Perhaps a more clear representation is:
{{{
imul    eax, [ebp+list+(ecx-1)*4]
}}}
You can set this in the manual dialog, but you'll have to uncheck the "Check operand" box, because IDA doesn't like this representation.

When faced with recursion, we want to first understand the base case; that is the case where the function returns without calling itself.
{{{
cmp     [ebp+arg_0], 1
}}}
In the recursive case, the function calls itself twice, and adds the results of the two. The calls subtract one and two respectively. The first:
{{{
mov     eax, [ebp+arg_0]
sub     eax, 1
push    eax
call    sub_401350
}}}
At this point we can track the first several cases:
*func(0) = 1
*func(1) = 1
*func(2) = func(1) + func(0) = 1 + 1 = 2
*func(3) = func(2) + func(1) = 2 + 1 = 3
*func(4) = func(3) + func(2) = 3 + 2 = 5

def func(x):
  if x < 1:

  return func(x-1) + func(x-2)

y = 0
z = 2
while(z < 0x37):
  y = func(z)
  print "%d: %d" % (z, y)

The Intel x86 Instruction set is complicated and [[well documented|http://www.intel.com/products/processor/manuals/index.htm]]. For brevity consider the following example: ''C7 45 F8 58 30 40 00''.
!!C7: The Opcode
The first byte(s) represent(s) the instruction to process; in this case: //mov//. The opcode my by several bytes long, though the most common instructions generally have only 1 or 2.
!45 F8: The ~ModR/M Byte and Displacement
The ~ModR/M byte (45) represents the first operand. This particular value represents the value pointed at by (EBP + a 1-byte displacement). In this case, the displacement F8, or -8.
!58 30 40 00: The 32-bit Immediate
The second operand is represented by a 4-byte immediate. Since x86 uses little-endian representation, the value is 0x00403058.

!! ~ServiceMain

The PEB contains a pointer to a PEB_LDR_DATA structure which 
{{{
typedef struct _PEB {
  BYTE Reserved1[2];
  BYTE BeingDebugged;
  ...
  PPEB_LDR_DATA Ldr; // offset: 0xC
  ...
}
}}}
The PEB_LDR_DATA structure contains head and tail pointers to three doubly linked loaded module lists:
#0x0C ~InLoadOrderModuleList
#0x14 ~InMemoryOrderModuleList
#0x1C ''~InInitializationOrderModuleList''
These lists refer to the same objects, most useful of which is the last one as the initialization order is predictable. First comes ntdll.dll, followed by kernel32.dll which contains both LoadLibrary and GetProcAddress. The base address of the module follows the ~InInitializationOrderModuleList LIST_ENTRY. Putting it all together:

{{{
xor   eax, eax                // Set eax = 0
mov   eax, fs:[eax+0x30]      // Locate the PEB
mov   eax, [eax + 0x0c]       // Retrieve PROCESS_MODULE_INFO pointer from the PEB
mov   esi, [eax + 0x1c]       // Get the address of flink (head) in the init module list (ntdll)
lodsd                         // Load the address of the next flink into eax (kernel32)
mov   eax, [eax + 0x8]        // Grab the module base address from the list entry
}}}

Every thread has an associated TEB (Thread environment Block). The TEB contains various book keeping data for the thread and refers to the PEB (Process Environment Block) which holds process wide meta-data. By convention the TEB is located at FS:[0] (or at the base of the Frame Segment). 48 (0x30) bytes into the TEB is a pointer to the PEB.

In C++, the ''this'' Pointer is a reference to the object that [[Methods]] act upon. This calling convention is commonly known as [[Thiscall]]. Understanding how to recognize the ''this'' pointer when reverse engineering object oriented code will is can aid in tracking objects and understanding the relationships between objects and their methods.

By convention, ''Microsoft Visual C++'' compilers pass the ''this'' pointer to instance methods through the ''ECX'' register, and passes the rest of the parameters onto the stack in reverse order (like [[Stdcall]]).

The GCC compiler ''Thiscall'' is based on [[Cdecl]]; however, it  pushes ''this'' onto the stack before the method call after all arguments have been pushed.

Some compilers, such as Borland and Watcom, reportedly store ''this'' in ''EAX''. As always, these are conventions, and though exceptions to the rules exist, they are fairly reliable.

When developing Classes in C++, inheritance is used to define characteristics and functionality only once for functionality shared by several related classes. For example, the following could be base class for people:
{{{
class Person {
protected:
    char name[50];
public:
    Person(char *_name) { strncpy(name, _name, 50); }
    virtual void work() {
        printf("%s moves some boxes\n", name);
    }
};
}}}
Here the Person class is defined with a single function which can be overridden any derived classes:
{{{
class Novelist : public Person {
public:
    Novelist(char *_name) : Person(_name) {}
    virtual void work() {
        printf("%s writes a book\n", name);
    }
};
}}}
A Novelist can be created using a Person pointer:
{{{
Person *Chaucer = new Novelist((char*)"Geoffrey Chaucer");
Chaucer->work();  // Will write "Geoffrey Chaucer writes a book\n" to STDOUT

When developers need to access system resources (e.g. disk, network), they typically use shared libraries provided with the operating system or third party software. In Windows, much of the core functionality is provided by ''kernel32.dll'' and other Dynamically Linked Libraries. In order to use these libraries, a program must load the library into its process space (LoadLibrary), and locate particular function addresses that are required (GetProcAddress).
There are multiple methods used to resolve dynamic library functions, three of which are:

Rather than force the OS to load functions at start up, libraries can be loaded on-demand using the LoadLibrary/GetProcAddress functions.
{{{
typedef int
(WINAPI *MessageBoxPtr)(
    HWND hWnd,
    LPCSTR lpText,
    LPCSTR lpCaption,
    UINT uType);
...
HMODULE hUser32 = LoadLibrary("User32.dll");
MessageBoxPtr pMessageBox = (MessageBoxPtr)GetProcAddress(hUser32, "MessageBoxA");
}}}
DLL files have a corresponding header file (.h) which describe the functions and global variables available in the library.
{{{
#include <Windows.h> // A catch all for many Windows libraries
}}}
The functions can be directly referenced and are loaded by the OS when the application is loaded into memory:
{{{
CreateFile(file_path, GENERIC_READ|GENERIC_WRITE, FILE_SHARE_READ, NULL, CREATE_NEW, FILE_ATTRIBUTE_NORMAL, NULL);
}}}
